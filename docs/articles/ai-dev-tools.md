# AI 開發工具功能說明

現代 AI 開發工具大多具備以下核心功能，理解這些概念有助於更有效地與任何 AI 工具協作。

- Model：所有功能的核心引擎，能力決定上限
- 行為規範文件 與 Memory：定義 AI 的工作規則與個人化偏好，是運作規範的基礎
- Commands：用一個指令觸發完整流程，減少重複輸入
- Skills：賦予 AI 可重複套用的專業執行能力
- MCP：讓 AI 突破邊界，主動連接外部世界
- Agent：自主規劃並執行任務，也可拆分給多個 AI 平行處理
- Context Window：AI 的工作記憶，理解其機制是管理大型任務的關鍵

這些功能都建立在 Model（AI 模型）之上——模型的能力決定了上限，工具層則決定你能發揮多少。彼此遞進，從奠定模型能力、定義規則、簡化操作、擴充能力，到連接外部世界、自主執行複雜任務。理解每層的用途，有助於在對的時機選用對的工具。

---

## Model（AI 模型）

一句話來說：Model 決定 AI 能做到多深、多準、多快。

Model 是所有功能的核心引擎，其他工具層都是建立在模型能力之上的延伸。選擇模型時，以下幾個面向會直接影響後續所有功能的表現：

- Context Window 大小：決定單次對話能處理多少內容，直接影響大型任務的可行性
- 工具呼叫（Tool Use）支援：Agent 和 MCP 功能仰賴模型能否正確呼叫外部工具，不是所有模型都支援
- 推理能力：影響 Agent 規劃任務、分解問題的品質
- 多模態支援：部分任務需要模型能處理圖片、文件等非文字輸入
- 費用與速度：模型能力越強，通常費用越高、回應越慢，需依使用情境取捨

不同平台通常會提供多家模型供應商的選項；以下分類較適合作為使用情境上的選型參考，而非各家模型供應商的官方分級。型號僅作為常見例子，實際名稱、能力與可用性仍以你所使用的平台為準：

高推理模型：

- 例如 GPT-5.4、Claude Opus、Gemini Pro 系列中的高階型號
- 通常適合大型重構、困難除錯、跨檔案分析與高難度技術決策

平衡型模型：

- 例如 Claude Sonnet 系列、GPT-5、部分平台提供的預設 coding 模型
- 通常適合日常開發、一般除錯、文件撰寫與大多數協作情境

快速型模型：

- 例如較輕量的 Claude Haiku、GPT mini、Gemini Flash 類模型
- 通常適合快速問答、小幅修改、語法查詢與重複性高的小任務

選擇建議：

- 不確定從哪開始：先從平衡型模型開始
- 深度推理或大型除錯：改用高推理模型
- 快速處理小修改或簡單提問：選快速型模型

要注意的是，不同平台上的模型名稱、可用性與預覽狀態常常會變動；同一篇文件提到的模型，不一定每位使用者都會在自己的介面中看到。

同一套工具設定，換用不同模型可能得到截然不同的結果。建議在使用 Agent、MCP 等進階功能前，先確認所選模型是否支援對應能力。

## 行為規範文件 與 Memory（記憶系統）

一句話來說：行為規範文件負責定義共同規則，Memory 負責累積持續可用的經驗與偏好。

行為規範文件是靜態的，定義團隊共同遵守的規則，每次對話皆生效。不同工具對應的檔名不同（如 Claude Code 用 `CLAUDE.md`、Cursor 用 `.cursorrules`、通用慣例用 `AGENTS.md`），但概念一致。

建議包含的內容：

- 專案概述
- 建置和測試命令
- 程式碼風格指南
- 測試說明
- 安全考量

Memory 是動態的，記住使用者偏好、專案慣例、常見解法等個人化資訊，跨對話持續沿用。各工具的累積方式不同，有些由 AI 自動寫入，有些需手動觸發。

使用時機：團隊共同規範寫入 AGENTS.md；個人化偏好或實作經驗交給 Memory 累積。

有了基礎規則之後，下一步是透過 Commands 將常用工作流程打包成指令，減少重複輸入。

---

## Commands（自訂指令）

一句話來說：Commands 解決的是「怎麼快速把需求說清楚」。

以 `/` 開頭的快捷指令，本質是預寫好的 prompt 模板，告訴 AI「要做什麼」。分為兩種：

1. 系統內建：工具預設提供的指令，如 `/help`、`/clear`
2. 自訂指令：由使用者或團隊自行定義，將複雜工作流程打包成一個指令，統一協作標準

例如 `/translate` 指令，背後的 prompt 模板可能是：「將以下內容翻譯成繁體中文，保留原有格式與專有名詞」，使用者只需輸入 `/translate` 即可觸發完整流程。

如何建立自訂指令：在專案的指令目錄下新增一個 Markdown 檔案，檔名即為指令名稱（如 `translate.md`），內容填入對應的 prompt 模板。不同工具的指令目錄位置不同，但概念一致——指令建立後，團隊成員皆可直接使用，確保執行標準一致。

Commands 解決「怎麼說」的問題；Skills 則進一步定義「怎麼做」，讓 AI 掌握可重複套用的專業執行方式。

---

## Skills（技能）

一句話來說：Skills 解決的是「AI 要用什麼專業方法做事」。

Skills 是封裝了指引、方法與資源的可重用執行模組，讓 AI 能以一致的方式完成特定類型的任務。

與外部連線工具不同，Skills 資料夾的 Resources 是靜態的本地資源（文件範本、規格說明、參考資料），搭配可在本地執行的腳本（Scripts），聚焦於本地端的任務執行，不依賴外部網路連線。

Skills 教導 AI 如何以可重複的方式完成特定任務，例如依照統一規範建立文件、使用特定工作流程分析資料，或自動化重複性任務。

Skills 分為兩種觸發方式：

- AI 自動判斷：執行任務過程中，AI 依據當前需求自行決定是否呼叫
- 使用者主動呼叫：部分 Skills 也支援以 `/` 指令直接觸發

Skills 也可以封裝特定領域的專業知識，啟動後 AI 會自動遵守對應的規範，無需在每次對話中重複說明：

- `/vue-best-practices`：以 Vue 3 Composition API 的最佳實踐撰寫程式碼
- `/laravel`：遵循 Laravel 的目錄結構、命名慣例、Eloquent 寫法
- `/frontend-design`：依特定設計原則生成 UI 元件
- `/tdd`：以測試驅動開發流程撰寫功能，先寫測試再實作
- `/code-review`：依統一標準對程式碼進行安全性、可維護性審查

如何建立 Skill：在工具指定的 Skills 目錄下新增一個資料夾，資料夾內放入對應的指令定義與資源檔案。不同工具的目錄位置與格式不同，但概念一致——建立後即可供 AI 動態載入使用。

```text
skill-name/
├── SKILL.md
├── scripts/          # 可執行程式碼
├── references/       # 需要時可查閱的其他文件
└── assets/           # 靜態資源
```

Skills 讓 AI 在工具箱內自主運作；MCP 則讓 AI 走出工具箱，直接操作外部世界。

## MCP（Model Context Protocol）

一句話來說：MCP 讓 AI 從只能聊天，變成能連接外部系統做事。

讓 AI 直接與外部服務互動的開放標準介面。沒有 MCP，AI 只能被動處理你貼進來的內容；有了 MCP，AI 可以主動查資料、操作其他系統，無需手動複製貼上。

MCP 除了提供可呼叫的工具（Tools）之外，也支援動態 Resources——由伺服器端即時產生的資料來源（如資料庫的即時查詢結果、外部 API 的當下回應），AI 可依需求主動讀取，每次取得的內容都反映當下狀態，無需事先準備靜態檔案。

常見 MCP Server：

- GitHub：操作 issue、PR、repository
- PostgreSQL / SQLite：查詢資料庫
- Slack：讀取訊息、發送通知
- Puppeteer：自動化瀏覽器操作
- Chrome DevTools：操控 Chrome 瀏覽器、擷取畫面、偵測網路請求
- Google Drive / Notion：讀取文件與筆記

注意事項：

- MCP Server 來源不一，非官方提供的第三方 Server 需特別謹慎，安裝前應確認來源可信、程式碼公開可審查
- 只開放必要的 MCP Server 存取權限，避免 AI 誤操作關鍵系統
- 敏感資料（如資料庫、私人文件）應確認 MCP Server 的讀寫權限設定是否符合預期
- MCP 呼叫通常會帶來額外的上下文與成本開銷；呼叫結果是否保留在對話歷史、以及如何計費，會依工具實作而異

當 MCP 提供了外部連接能力後，搭配 Agent 的自主規劃能力，AI 便能完成端到端的複雜任務。

---

## Agent（智能代理）

一句話來說：Agent 是把「思考」和「執行」串起來的自動化工作者。

Agent 是能自主規劃並依序執行多步驟任務的 AI。給定一個目標後，Agent 會依工具能力與權限範圍，自行決定該做什麼、用哪些工具、依照什麼順序執行。

在部分支援 agent 的工具中，系統可能會將複雜任務拆分成多個子代理（subagent）平行或串行處理；是否支援、自動化程度，以及上下文如何隔離，會依平台實作而異。

部分工具支援為 Agent 或各子代理指定不同的模型，例如讓規劃階段使用推理能力較強的模型，執行階段改用速度較快、成本較低的模型，依任務需求自由搭配。

常見的 Agent 類型：

- Plan agent：專門負責規劃實作步驟，分析架構與拆解任務，不直接寫程式碼
- 研究 agent：負責蒐集資料、搜尋文件
- 執行 agent：負責實際寫程式碼、修改檔案、執行測試

適合的使用情境：

- 大型重構：拆分成多個獨立模組同時處理
- 研究與實作分離：各 agent 各司其職，不互相干擾
- 平行測試：多個子代理同時對不同功能執行測試

注意事項：

- Agent 的自主性越高，誤操作的風險也越高，建議在任務描述中明確指定邊界（如「只讀取，不修改」）
- 執行大型任務前，先確認 AI 的規劃步驟是否合理
- 每個子代理都會消耗獨立的 token 配額，複雜任務的成本需事先評估

所有 Agent 的運作都受限於 Context Window，理解其機制是管理大型任務的關鍵。

## Context Window（上下文）

一句話來說：上下文決定 AI 一次能記住多少資訊。

Context Window（上下文）是 AI 在單次對話中能處理的最大內容量，包含你輸入的所有訊息、AI 的回覆、讀取的檔案內容等。超出上限後，AI 可能無法存取較早的內容，或由工具自動進行摘要壓縮。

### Context 的組成

一個完整的 Context Window 通常由以下幾個部分構成：

- System Prompt：行為規範文件（如 CLAUDE.md）、Skills 載入的指引，以及工具的定義說明；在每次對話開始時就已佔用固定的 token 配額
- Tools（工具呼叫）：AI 每次呼叫工具的請求與工具回傳的結果，均會計入 context；任務涉及多輪工具呼叫時，這部分的 token 消耗會持續累積
- User Messages（對話歷史）：每一輪對話中使用者的輸入與 AI 的回覆，隨對話進行逐步累積
- Resources：MCP Server 提供的即時資料、本地 Skills 的靜態資源、讀入的檔案內容，以及其他注入的參考資訊

理解各部分的佔比，有助於在遇到 context 不足時，針對性地進行清理或壓縮。

常見的影響：

- 對話太長導致 AI 忘記最初的需求
- 讀入太多檔案後，早期的上下文被擠掉
- 大型任務難以在單次對話中完成

管理方式：

- 切換任務：執行 `/new` 開啟全新對話，確保 context 乾淨不受前一個任務干擾
- 拆分任務：將大任務拆成多個小任務分開執行
- 使用 subagent：讓子代理處理子任務，各自有獨立的 context
- 自動壓縮：大多數工具會在接近上限時自動摘要對話歷史，也可手動執行 `/compact` 提前釋放空間
- 查看用量：部分工具會提供查看 context 使用狀況的指令或介面，包含各類別佔用的 token 數
